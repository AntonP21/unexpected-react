    to contain

    • <RenderedReactElement> [not] to contain [with all children] [with all wrappers] <ReactElement|string>
    • <RenderedReactElement> [not] to contain [exactly] <ReactElement|string>

    It's possible to check for a part of the subtree, without testing the entire returned tree. This allows you to test specific elements, without writing brittle tests that break when the structure changes.

    Assuming the following component output

    class MyComponent extends React.Component {
      render () {
        return (
          <div>
            <span className="top">one</span>
            <span className="middle">two</span>
            <span className="bottom">three</span>
          </div>
        );
      }
    };
     
    var component = TestUtils.renderIntoDocument(
      <MyComponent/>
    );
    // This will pass, as `<span className="middle">two</span>` can be found in the component's output
    expect(component, 'to contain', <span>two</span>);

    Notice that the extra className="middle" in the <span className="middle">two</span> is ignored, in a similar way to the to have rendered assertion.

    You can override this behaviour by using 'to contain exactly', and 'to contain with all children'

    // This will fail, as `<span>two</span>` cannot be found in the renderers output, due to
    // the missing `className="middle"` prop
    expect(component, 'to contain exactly', <span>two</span>);
    expected undefined to contain exactly <span>
    two
    </span>
      No matching assertion, did you mean:
      <ReactElement> [not] to contain [exactly] <ReactElement|string>
      <ReactPendingDeepEvent> [not] to contain [exactly] <ReactElement>
      <ReactPendingShallowEvent> [not] to contain [exactly] <ReactElement>
      <ReactShallowRenderer> [not] to contain [exactly] <ReactElement|string>
      <RenderedReactElement> [not] to contain [exactly] <ReactElement|string>
      <RenderedReactElementData> [not] to contain [exactly] <ReactElement|string>

    The same thing applies to children for 'to contain' as for 'to have rendered'.
